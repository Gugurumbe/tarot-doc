Regroupe les traitements du protocole.

Utilité : le type de message "atout" est en concurrence avec la définition ATOUT pour la couleur des atouts.

Valeurs énumérées
ERREUR_PROTOCOLE 

Transmission mixte :

  1. client -> serveur : le client signale qu'il n'a pas compris le message du serveur. Risque d'être ignoré par le serveur.
  2. serveur -> client : le serveur informe le client qu'il ne respecte pas le même protocole que lui, ou qu'il essaye de jouer sans que ce soit son tour.
    Voir également
    Msg_erreur_protocole
REFUSE 

Transmission serveur -> client : la requête du client a été refusée. Il n'a pas la bonne Carte, il a appelé une Dame alors qu'il n'a pas tous les rois, il met un bout au chien, ou une autre tentative de triche.

Voir également
Msg_refuse
PartieClient::action_refusee()
NUMERO 

Transmission serveur -> client : le serveur donne son numéro de tour au client.

Voir également
Msg_numero
DISTRIBUTION 

Transmission serveur -> client : le serveur distribue 15 cartes au client.

Voir également
Msg_distribution
PRISE 

Transmission client -> serveur : le client dit s'il prend.

Voir également
Msg_prise
CONTRAT 

Transmission serveur -> client : le serveur informe les 5 clients de la prise du précédent joueur.

Voir également
Msg_contrat
APPEL 

Transmission serveur -> client : le serveur informe le client qui a la plus grosse Enchère qu'il doit appeler une carte. Ainsi, le client n'a pas à vérifier s'il a l'enchère maximale.

Voir également
Msg_appel
APPELER 

Transmission client -> serveur : le client informe le serveur qu'il veut écarter cette carte.

Voir également
Msg_appeler
CONTRAT_FINAL 

Transmission serveur -> client : le serveur informe tous les clients du contrat final.

Voir également
Msg_contrat_final
CHIEN 

Transmission facultative serveur -> client : si la prise n'est ni Garde Sans ni Garde Contre, on montre le chien à tous. Le preneur se doit d'effectuer un écart.

Voir également
Msg_chien
ECART 

Transmission facultative client -> serveur : si la prise n'est ni Garde Sans ni Garde Contre, le client fait une proposition des 3 cartes qu'il veut écarter.

Voir également
Msg_ecart
ATOUT 

Transmission facultative serveur -> client : si un atout est présent dans l'écart, il doit être montré.

Voir également
Msg_atout
CHELEM 

Transmission client -> serveur nécessaire : le client décide s'il demande un chelem ou pas.

Avertissement
Ce type de Message va bientôt être supprimé, car seul le preneur doit décider s'il demande un chelem ou pas.
Voir également
Msg_chelem
JEU 

Transmission serveur -> client : le serveur annonce que le jeu commence, et éventuellement s'il y a un chelem déclaré.

Voir également
Msg_jeu
MONTRER_POIGNEE 

Transmission client -> serveur : le client tente de montrer une poignée. Elle doit être validée par le serveur.

Voir également
Msg_montrer_poignee
POIGNEE 

Transmission serveur -> client : le serveur informe tous les clients que le joueur dont c'est encore le tour a déclaré une poignée.

Voir également
Msg_poignee
REQUETE 

Transmission client -> serveur : le client demande à jouer une certaine Carte.

Voir également
Msg_requete
CARTE 

Transmission serveur -> client : le joueur dont c'était le tour a joué cette Carte.

Voir également
Msg_carte
PLI 

Transmission serveur -> client : c'est tel joueur qui remporte le pli (c'est donc à lui de jouer). Envoyé juste après le Msg_carte.

Voir également
Msg_pli
RESULTAT 

Transmission serveur -> client : voici le résultat de cette partie.

Voir également
Msg_resultat
